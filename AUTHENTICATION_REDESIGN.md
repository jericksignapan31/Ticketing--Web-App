# Authentication Redesign - Change Log (May 22, 2026)

## 📋 Overview
Transformed the authentication system from **employee_id-based usernames** to **email-based usernames** for a more user-friendly and intuitive login experience.

---

## 🔄 Changes Made

### 1. **SignupDto** - `src/auth/dto/signup.dto.ts`
**What Changed:**
- ❌ **Removed** `employee_id` field requirement
- ✅ **Kept** `email` field as primary username identifier
- Updated API documentation

**Before:**
```typescript
@IsNotEmpty()
employee_id: string;  // Required

@IsEmail()
@IsNotEmpty()
email: string;  // Secondary field
```

**After:**
```typescript
// employee_id field completely removed

@IsEmail()
@IsNotEmpty()
email: string;  // Now primary identifier for username
```

**Impact:** Signup now requires email only, not employee_id

---

### 2. **LoginDto** - `src/auth/dto/login.dto.ts`
**What Changed:**
- Updated example in API documentation
- Changed username example from generic "admin" to email format

**Before:**
```typescript
@ApiProperty({ example: 'admin', description: 'Username' })
username: string;
```

**After:**
```typescript
@ApiProperty({ example: 'john.doe@example.com', description: 'Email (used as username)' })
username: string;
```

**Impact:** Users now understand they should use their email to login

---

### 3. **AuthService.signup()** - `src/auth/auth.service.ts`
**What Changed:**
- Changed uniqueness validation from `employee_id` to `email`
- Use `email` as username when creating `UserAccount`
- Employee `employee_id` now auto-generated as UUID (no longer required input)

**Before:**
```typescript
// Check if employee_id already exists
const existingEmployee = await this.employeeRepository.findOne({
  where: { employee_id: signupDto.employee_id },
});

// Create employee with provided employee_id
const employee = manager.create(Employee, {
  employee_id: signupDto.employee_id,
  ...
});

// Create user account with employee_id as username
const userAccount = manager.create(UserAccount, {
  employee_id: signupDto.employee_id,
  username: signupDto.employee_id,  // ❌ OLD: employee_id as username
  password: hashedPassword,
});
```

**After:**
```typescript
// Check if email already exists
const existingEmail = await this.employeeRepository.findOne({
  where: { email: signupDto.email },
});

// Create employee with auto-generated employee_id
const employee = manager.create(Employee, {
  // employee_id auto-generated by @PrimaryGeneratedColumn('uuid')
  email: signupDto.email,
  ...
});

// Create user account with email as username
const userAccount = manager.create(UserAccount, {
  employee_id: savedEmployee.employee_id,
  username: signupDto.email,  // ✅ NEW: email as username
  password: hashedPassword,
});
```

**Impact:** Signup now uses email as unique identifier and username

---

### 4. **AuthService.login()** - `src/auth/auth.service.ts`
**What Changed:**
- No functional changes - already queries by username (now email)
- JWT token payload unchanged - still includes `employeeId`, `username`, etc.

**Note:** The login method works automatically because:
- It queries `where { username }` and username is now the email
- JWT payload still contains full user context

**Impact:** Login seamlessly transitions to email-based credentials

---

### 5. **Employee Entity** - `src/entities/employee.entity.ts`
**What Changed:**
- Changed `employee_id` from manual `@PrimaryColumn()` to auto-generating `@PrimaryGeneratedColumn('uuid')`
- Removed `IsBoolean` import (unused)

**Before:**
```typescript
@PrimaryColumn()
employee_id!: string;  // Manual entry required
```

**After:**
```typescript
@PrimaryGeneratedColumn('uuid')
employee_id!: string;  // Auto-generated UUID
```

**Impact:** Employee IDs are now automatically generated UUIDs instead of manual strings

---

### 6. **UserAccount Entity** - `src/entities/user-account.entity.ts`
**What Changed:**
- Fixed column name mapping - `password` column maps to database `password` column (was incorrectly mapped to `password_hash`)

**Before:**
```typescript
@Column({ name: 'password_hash' })
password!: string;  // ❌ Incorrect database column
```

**After:**
```typescript
@Column()
password!: string;  // ✅ Correct database column
```

**Impact:** Password hashing/comparison now works correctly with database

## 🔐 Authentication Flow

### Old Flow (Employee ID Based)
```
User Registration:
  Input: employee_id="EMP001", email="user@company.com"
  ↓
  Creates: Employee(id="EMP001"), UserAccount(username="EMP001")
  
User Login:
  Input: username="EMP001", password="secret"
  ↓
  Finds UserAccount where username="EMP001"
  ↓
  Returns JWT with sub="EMP001"
```

### New Flow (Email Based)
```
User Registration:
  Input: email="user@company.com" (no employee_id required)
  ↓
  Creates: Employee(id=<auto-uuid>), UserAccount(username="user@company.com")
  
User Login:
  Input: username="user@company.com", password="secret"
  ↓
  Finds UserAccount where username="user@company.com"
  ↓
  Returns JWT with sub=<user_id>, username="user@company.com", employeeId=<uuid>
```
